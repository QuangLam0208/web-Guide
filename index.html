HTTP Supplementary Guide — Filling the Gaps for Beginners

This document supplements the existing HTTP guide with the most important topics that were missing or underdeveloped. Each section is written for absolute beginners learning web development.

SUPPLEMENT 1 — Seeing HTTP in Real Life: Chrome DevTools

One of the most important skills for any web developer is being able to see real HTTP requests and responses in your browser. You do not need any special tool — it is built into Chrome (and all modern browsers).

How to Open DevTools Network Tab

  1. Open Google Chrome.
  2. Visit any website, for example: https://example.com
  3. Press F12 on Windows/Linux, or Cmd + Option + I on Mac.
  4. Click the "Network" tab at the top of the DevTools panel.
  5. Refresh the page (Ctrl + R or Cmd + R).
  6. You will now see every HTTP request the page made.

What You Will See

Name              Status  Method  Type       Size    Time
example.com/      200     GET     document   2.4 kB  210 ms
style.css         200     GET     stylesheet 1.1 kB  45 ms
logo.png          200     GET     image      8.3 kB  78 ms

Each row is one HTTP request.

Clicking a Request — What Is Inside

When you click on any row, you will see sub-tabs:

Headers tab — shows the request headers you sent and the response headers the server sent back.

Request Headers:
  GET / HTTP/1.1
  Host: example.com
  User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64) ...
  Accept: text/html,application/xhtml+xml,...

Response Headers:
  HTTP/1.1 200 OK
  Content-Type: text/html; charset=UTF-8
  Content-Length: 1256
  Cache-Control: max-age=86400

Preview tab — shows the formatted response body (HTML rendered, JSON structured).

Response tab — shows the raw text of the response body.

Timing tab — shows exactly how long each phase took (DNS lookup, TCP connect, server processing, download).

Why This Matters

Reading DevTools is how real developers:

  • Debug why an API call is failing.
  • Check what headers are being sent.
  • Verify that authentication tokens are attached.
  • Inspect what data the server actually returned.

Practice: Open DevTools, visit https://jsonplaceholder.typicode.com/posts/1, and look at the Network tab. You will see a real API call with a real JSON response.

SUPPLEMENT 2 — Query Parameters vs Path Parameters

When you request a resource from a server, there are two common ways to pass extra information in the URL.

Path Parameters

Path parameters are part of the URL path itself. They identify a specific resource.

GET /users/42
GET /products/laptop
GET /orders/8899

In each case, 42, laptop, and 8899 are path parameters. They tell the server which specific item you want.

Think of it like a street address: /users/42 means "give me the specific user who lives at position 42".

Query Parameters

Query parameters come after a ? in the URL. They are used for filtering, sorting, searching, or optional settings.

GET /products?category=laptop&sort=price&page=2

Breaking this down:

PartMeaning
/productsThe base resource
?Marks the start of query params
category=laptopFilter by category "laptop"
&Separator between params
sort=priceSort results by price
page=2Show the second page of results

Multiple Query Parameters

You can chain as many as you need using &:

GET /flights?from=HAN&to=SGN&date=2025-12-01&class=economy

When to Use Which?

SituationUseExample
You want a specific itemPath parameterGET /users/5
You want to filter a listQuery parameterGET /users?role=admin
You want to searchQuery parameterGET /products?q=headphones
You want to sort resultsQuery parameterGET /posts?sort=date&order=desc
You want a nested resourcePath parameterGET /users/5/orders

Real-World Example: YouTube

https://www.youtube.com/watch?v=dQw4w9WgXcQ

Here, watch is the path, and v=dQw4w9WgXcQ is a query parameter — the video ID. YouTube uses a query param here because watch is not a resource name; it is an action.

Encoding Special Characters

Query parameter values cannot contain spaces or certain special characters. They must be URL-encoded:

Space   →  %20  or  +
&       →  %26
=       →  %3D

Example:
GET /search?q=hello%20world
GET /search?q=hello+world

Modern browsers and HTTP libraries handle this encoding automatically.

SUPPLEMENT 3 — Request Body Formats (Content-Type in Depth)

When you send a POST, PUT, or PATCH request, you often include a body — the data you want to send to the server. There are several different formats for that body, and the server needs to know which format you are using.

The Key Rule

If you send a body, you must tell the server its format using the Content-Type request header.

This is one of the most common causes of bugs for beginners. If you send JSON but forget to set Content-Type: application/json, the server receives the data but does not know how to read it. The result is usually a 400 Bad Request error.

Format 1: JSON (application/json)

The most common format for modern APIs.

Request header:

Content-Type: application/json

Request body:

{
  "name": "Alice",
  "email": "alice@example.com",
  "age": 25
}

Used for: REST APIs, sending structured data between frontend and backend.

Format 2: Form Data (application/x-www-form-urlencoded)

This is what a traditional HTML <form> submits by default. Data is encoded as key-value pairs, similar to query parameters.

Request header:

Content-Type: application/x-www-form-urlencoded

Request body (raw text, not JSON):

name=Alice&email=alice%40example.com&age=25

Used for: HTML form submissions, some older APIs, login forms.

Format 3: Multipart Form Data (multipart/form-data)

Used when you need to upload files along with other data. Each piece of data is a separate "part" in the request.

Request header:

Content-Type: multipart/form-data; boundary=----FormBoundary7MA4

Request body:

------FormBoundary7MA4
Content-Disposition: form-data; name="username"

alice
------FormBoundary7MA4
Content-Disposition: form-data; name="avatar"; filename="photo.jpg"
Content-Type: image/jpeg

[binary image data here]
------FormBoundary7MA4--

Used for: File uploads, profile picture changes, document submissions.

Format 4: Plain Text (text/plain)

Sends raw text, rarely used in APIs.

Request header:

Content-Type: text/plain

Request body:

Hello server, this is plain text.

Comparison Table

FormatHeader ValueBest Used For
JSONapplication/jsonREST APIs, structured data
URL-encoded formapplication/x-www-form-urlencodedSimple HTML forms, login
Multipart form (with files)multipart/form-dataFile uploads
Plain texttext/plainSimple text payloads

The Content-Type ↔ Body Connection (Very Important)

The Content-Type header and the body format must always match. If they do not, the server will reject or misread your request.

WRONG ❌ — Header says JSON but body is form-encoded
Content-Type: application/json
Body: name=Alice&age=25

CORRECT ✅ — Header matches the body format
Content-Type: application/json
Body: {"name": "Alice", "age": 25}

WRONG ❌ — Header says form-encoded but body is JSON
Content-Type: application/x-www-form-urlencoded
Body: {"name": "Alice", "age": 25}

CORRECT ✅ — Header matches the body format
Content-Type: application/x-www-form-urlencoded
Body: name=Alice&age=25

SUPPLEMENT 4 — Response Headers (Explained for Beginners)

The existing guide covered request headers well, but the server also sends headers back in the response. These response headers are equally important to understand.

Common Response Headers

HeaderWhat It DoesExample Value
Content-TypeTells the client what format the response body is inapplication/json
Content-LengthHow many bytes are in the response body1256
Content-EncodingHow the body is compressedgzip
Cache-ControlHow long the client should cache this responsemax-age=3600
Set-CookieCreates a cookie in the browsersessionId=abc123; HttpOnly

Retry-AfterHow long to wait before making the next request30

The Content-Type Response Header

Just like in requests, the server sets Content-Type in the response so the client knows how to parse the body.

HTTP/1.1 200 OK
Content-Type: application/json

{"user": "Alice", "id": 42}

If the response Content-Type is text/html, the browser renders it as a webpage. If it is application/json, JavaScript can parse it as data. If it is image/png, the browser displays it as an image.

The Set-Cookie Header

This is how the server plants a cookie in your browser. The next time you visit, the browser automatically sends that cookie back.

HTTP/1.1 200 OK
Set-Cookie: sessionId=abc123; HttpOnly; Secure; Max-Age=86400
OptionMeaning
HttpOnlyJavaScript cannot read this cookie (prevents XSS attacks)
SecureCookie is only sent over HTTPS connections
Max-AgeCookie expires after this many seconds
SameSiteControls when cookie is sent on cross-site requests

The Location Header (Redirects)

When the server responds with a 301 or 302 status code, the Location header tells the browser where to go next.

HTTP/1.1 301 Moved Permanently
Location: https://www.example.com/new-url

The browser automatically follows this redirect. You usually see this when websites move pages or redirect HTTP to HTTPS.

SUPPLEMENT 5 — Annotated Raw HTTP Messages

HTTP is just plain text sent over a network connection. Understanding what the raw message looks like removes a lot of mystery.

A Complete Raw HTTP Request

POST /api/login HTTP/1.1                  ← Request Line: method + path + version
Host: example.com                         ← Which server to talk to
Content-Type: application/json            ← The format of the body we are sending
Content-Length: 46                        ← How many bytes the body is
Accept: application/json                  ← What format we want back
User-Agent: MyApp/1.0                     ← Who we are (app name and version)
Authorization: Bearer eyJhbGci...         ← Our authentication token

  "email": "alice@example.com",
  "password": "secretpassword"
}

SUPPLEMENT 6 — Understanding 401 vs 403 (Common Beginner Confusion)

These two status codes are often confused because their names are counterintuitive in everyday English.

The Core Difference

CodeNameWhat It Actually Means
401 UnauthorizedShould be called "Unauthenticated"You have not proven who you are yet. No valid credentials were provided.
403 ForbiddenCorrectly namedYou have proven who you are, but you are not allowed to access this resource.

Simple Analogy

Imagine a company office building:

  • 401 = You tried to enter the building without a badge. The security guard says: "I don't know who you are. Show me your ID."
  • 403 = You showed your badge and the guard recognizes you as an intern. But you are trying to access the executive floor. The guard says: "I know exactly who you are, but you don't have permission to go there."

Code Examples

Scenario 1: No token provided → 401

GET /api/profile HTTP/1.1
Host: example.com
(no Authorization header)

Response:
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api"

{"error": "Authentication required. Please log in."}

Scenario 2: Invalid or expired token → 401

GET /api/profile HTTP/1.1
Authorization: Bearer expired_or_invalid_token

Response:
HTTP/1.1 401 Unauthorized

{"error": "Token is invalid or has expired."}

Scenario 3: Valid token, but insufficient permissions → 403

DELETE /api/users/5 HTTP/1.1
Authorization: Bearer valid_regular_user_token

Response:
HTTP/1.1 403 Forbidden

{"error": "Only administrators can delete users."}

Quick Debugging Guide

When you see 401, ask yourself:

  • Did I forget to send the Authorization header?
  • Is my token expired? Do I need to refresh it or log in again?
  • Is the token format correct? (e.g., Bearer prefix required?)

When you see 403, ask yourself:

  • Am I logged in as the right user? (Maybe test with an admin account)
  • Does my account have the right role or permission?
  • Is this resource restricted to certain plans or tiers?

SUPPLEMENT 7 — What to Do When You See HTTP Errors

Status codes are not just numbers to memorize — you need to know how to react when you encounter them. Here is a practical guide for beginners.

400 Bad Request

What happened: You sent something the server could not understand or accept.

Common causes:

  • Missing required fields in the JSON body
  • Sending the wrong data type (string instead of number)
  • Malformed JSON (missing a comma, extra bracket, etc.)
  • Wrong Content-Type header

What to do:

  1. Check the response body — the server usually explains what is wrong (e.g., "email is required")
  2. Validate your JSON is syntactically correct (use a JSON validator tool)
  3. Verify all required fields are present
  4. Check the Content-Type header matches your body format

401 Unauthorized

What happened: You are not authenticated.

What to do:

  1. Make sure you are sending the Authorization header
  2. Check if your token has expired — log in again to get a fresh token
  3. Verify the token format: most APIs want Authorization: Bearer your_token_here
  4. Check if the token is being read from the right place in your code

403 Forbidden

What happened: You are authenticated but not allowed.

What to do:

  1. Verify you are using the correct account (not a test account that lacks permissions)
  2. Check if the endpoint requires a special role (admin, moderator, paid plan)
  3. Read the API documentation for permission requirements

404 Not Found

What happened: The resource does not exist at that URL.

Common causes:

  • Typo in the URL
  • The resource was deleted
  • Wrong base URL (e.g., staging URL in production code)
  • You are using an ID that does not exist

What to do:

  1. Double-check the URL carefully for typos
  2. Confirm the resource ID actually exists
  3. Check the API documentation to verify the correct endpoint path
  4. Make sure you are pointing at the right environment (development vs production)

422 Unprocessable Entity

What happened: The server understood the request format, but the data fails validation rules.

Example: You sent valid JSON, but the age field is -5, which makes no business sense.

What to do:

  1. Read the response body carefully — the server usually lists exactly which fields failed and why
  2. Fix the data to meet the business rules (valid age range, required email format, etc.)

429 Too Many Requests

What happened: You have sent too many requests in a short time. The server is rate-limiting you.

What to do:

  1. Check the Retry-After response header — it tells you how many seconds to wait
  2. Check the X-Rate-Limit-Remaining header if present
  3. Add delays between your requests
  4. Implement exponential backoff in your code

500 Internal Server Error

What happened: Something broke on the server. This is not your fault.

What to do:

  1. Check if the service is down (visit the provider's status page)
  2. Try again later
  3. If building your own API: check your server logs for the actual error
  4. Report the issue to the API provider if it persists

502 Bad Gateway / 503 Service Unavailable

What happened: The server (or a proxy in front of it) is down or overloaded.

What to do:

  1. Wait and retry — these are usually temporary
  2. Check the service's status page

CORS Error (Not an HTTP Status Code, but Very Common)

What happened: Your browser blocked the request because the server did not allow cross-origin requests.

You will see something like this in the console:

Access to fetch at 'https://api.example.com/data' from origin
'http://localhost:3000' has been blocked by CORS policy.

What to do (as a frontend developer):

  1. This is a backend configuration problem, not a frontend one
  2. Tell your backend developer to add the Access-Control-Allow-Origin header
  3. As a temporary fix during development, you can use a CORS proxy

What to do (as a backend developer): Add these headers to your server response:

Access-Control-Allow-Origin: http://localhost:3000
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization

SUPPLEMENT 8 — HTTP Persistent Connections Explained

The Problem HTTP/1.0 Had

In the original HTTP/1.0, every single request required:

  1. Open a new TCP connection (this involves a "handshake" that takes time)
  2. Send the HTTP request
  3. Receive the HTTP response
  4. Close the TCP connection

For a simple webpage that loads 30 images, CSS files, and JavaScript files, this meant 30 separate TCP connections being opened and closed. Each connection takes extra time (typically 50–200 ms for a round trip). This was extremely slow.

HTTP/1.1 Solution: Keep-Alive (Persistent Connections)

HTTP/1.1 introduced persistent connections (also called keep-alive). The TCP connection stays open after a request, allowing multiple requests to reuse the same connection.

HTTP/1.0 (old, slow):
  Open TCP → GET index.html → Close TCP
  Open TCP → GET style.css  → Close TCP
  Open TCP → GET logo.png   → Close TCP

HTTP/1.1 (persistent):
  Open TCP
    → GET index.html
    → GET style.css
    → GET logo.png
  Close TCP (only when done or after timeout)

This alone made web pages significantly faster.

HTTP/2 Solution: Multiplexing

HTTP/1.1 persistent connections still had a limitation: requests had to be sent one at a time on a connection (you had to wait for a response before sending the next request, a problem called "head-of-line blocking").

HTTP/2 solved this with multiplexing — multiple requests can be in flight simultaneously over a single TCP connection, like multiple phone calls happening at the same time over the same wire.

HTTP/2 (multiplexed over one TCP connection):
  Open TCP
    → GET index.html (stream 1)
    → GET style.css  (stream 2)  ← sent at the same time, no waiting!
    → GET logo.png   (stream 3)  ← sent at the same time!
    ← Response for stream 2 (arrived first, out of order — that is fine)
    ← Response for stream 1
    ← Response for stream 3
  Close TCP

HTTP/3 Solution: QUIC

HTTP/3 goes further by replacing TCP entirely with a new protocol called QUIC, which is built on top of UDP.

Why? Because even with multiplexing, if a single TCP packet is lost, all streams have to wait for that packet to be retransmitted (TCP's own head-of-line blocking). QUIC solves this — a lost packet only blocks the specific stream it belongs to, not all streams.

VersionTransportConnectionsPerformance
HTTP/1.0TCP1 per requestSlow
HTTP/1.1TCPPersistent, but one-at-a-timeBetter
HTTP/2TCPMultiplexed streamsFast
HTTP/3QUIC/UDPMultiplexed, no head-of-lineFastest

SUPPLEMENT 9 — Full End-to-End Worked Example

This is the most important section. Let us trace every step of what happens when a JavaScript frontend application fetches a user profile from an API.

The Scenario

A React app running at http://localhost:3000 wants to fetch the profile of user ID 42 from an API at https://api.example.com.

The frontend code:

const response = await fetch('https://api.example.com/users/42', {
  method: 'GET',
  headers: {
    'Authorization': 'Bearer eyJhbGciOiJIUzI1NiJ9.abc',
    'Accept': 'application/json'
  }
});
const user = await response.json();

Step 1: URL Parsing

The browser parses https://api.example.com/users/42:

PartValue
Protocolhttps
Hostnameapi.example.com
Port443 (default for HTTPS)
Path/users/42

Step 2: DNS Lookup

The browser needs to find the IP address of api.example.com.

Browser asks DNS resolver: "What is the IP for api.example.com?"
DNS resolver responds:      "It is 203.0.113.42"

This lookup may be instant if the result is cached, or take ~20–100 ms if not cached.

Step 3: TCP + TLS Handshake

The browser opens a TCP connection to 203.0.113.42 on port 443.

Because this is HTTPS, a TLS (encryption) handshake also occurs:

  1. Browser says: "I want to connect securely."
  2. Server sends its SSL certificate.
  3. Browser verifies the certificate is valid and from a trusted authority.
  4. An encryption key is agreed upon.
  5. All further communication is encrypted.

This takes approximately 100–300 ms for a new connection (much less for subsequent requests on the same connection).

Step 4: HTTP Request Is Sent

The browser sends this raw HTTP request (encrypted via TLS, but conceptually this is the text):

GET /users/42 HTTP/2
Host: api.example.com
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.abc
Accept: application/json
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36

Note: no body is needed because GET requests do not carry a body.

Step 5: Server Processes the Request

The server at api.example.com receives the request and:

  1. Routes /users/42 to the correct handler (a function that handles user profile requests)
  2. Extracts 42 as the user ID from the path
  3. Validates the Authorization header — checks the JWT token
  4. Looks up user ID 42 in the database
  5. Serializes the result to JSON

Total server processing time: typically 10–100 ms depending on database queries.

Step 6: HTTP Response Is Sent

The server sends back:

HTTP/2 200 OK
Content-Type: application/json; charset=utf-8
Content-Length: 127
Cache-Control: private, max-age=60
X-Request-Id: req_f8a2c1d0e3b5

{
  "id": 42,
  "name": "Alice Nguyen",
  "email": "alice@example.com",
  "role": "developer",
  "joined": "2024-03-15"
}

Step 7: Browser/Client Processes the Response

The JavaScript fetch() receives the response:

  1. response.status is 200 — success.
  2. response.headers.get('Content-Type') is application/json.
  3. response.json() parses the body into a JavaScript object.
  4. The React component receives the user object and re-renders with the data.

Step 8: The Browser May Request Additional Assets

If the response included links to images, CSS, or other resources, the browser issues additional HTTP requests for each one — but now reusing the same TCP connection (because HTTP/2 multiplexing allows it).

Step 9: Complete Timeline Visualization

0ms       DNS Lookup
20ms      DNS resolved → IP: 203.0.113.42
20ms      TCP Connect
40ms      TCP established
40ms      TLS Handshake begins
120ms     TLS Handshake complete (connection secure)
120ms     HTTP Request sent:  GET /users/42
145ms     Server receives request, starts processing
200ms     Database query complete
210ms     Server sends HTTP Response: 200 OK
230ms     Browser receives full response
230ms     response.json() parses body
231ms     React re-renders with user data

Total time: ~231 ms (this is realistic for a cross-region API call).

Quick Reference Card

The Complete HTTP Request/Response Checklist

When debugging any HTTP issue, check these in order:

Request side:

  • Is the URL correct (no typos, correct domain, correct path)?
  • Is the HTTP method correct (GET vs POST vs PUT vs DELETE)?
  • Does the Content-Type header match the body format?
  • Is the Authorization header present and correctly formatted?
  • Is the request body valid JSON (if sending JSON)?
  • Are all required fields included in the body?

Response side:

  • What is the status code? (See status code guide above)
  • What does the response body say? (It usually explains the error)
  • Is there a CORS error in the browser console?
  • Is the Content-Type in the response what you expected?

Network side:

  • Can you reach the server at all? (Try opening the URL in a browser)
  • Is it a DNS issue? (Try ping api.example.com in terminal)
  • Is it a certificate issue? (Check the browser's security warning)
